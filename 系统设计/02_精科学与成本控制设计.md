# 法硕定制系统 · 精 / 科学 / 成本可控 三角设计

> 生成日期：2026-06-05（第 3 轮）
> 前提：云已决策**不用 Tailscale**，背诵与答疑都自建、都做到最精最科学、成本可控（P0-1 关闭）。
> 本文档回答一个问题：**自建双核，怎么同时做到"精 + 科学 + 省"？**

---

## 0. 一句话核心洞察

> **v2.3 强制 grep 这件事，本身就是省钱的钥匙。**

PC 端 Claude Code 答疑时，并不是把 8 万字心得 + 教材全文"读"进脑子，而是 **grep 出相关片段**再答。
所以 Web 端答疑根本不需要"全量注入 8-10 万 token"——那既烧钱又破缓存。
正确做法 = **规则全量常驻（稳定缓存）+ 知识按需检索（grep 工具拉片段）**。
这一个设计同时解决：成本（P1-1）、缓存矛盾（P0-3）、且**比全量注入更忠于 v2.3 的精神**（精）。

---

## 1. 成本可控：三根杠杆

成本 = 调用次数 × 每次 token 量 × 单价（按模型）。三根杠杆各个击破。

### 杠杆 A：上下文从"全量注入"改为"规则常驻 + 知识检索"（每次 token 量 ↓80%）

| 层 | 内容 | 注入方式 | 是否进缓存前缀 |
|---|---|---|---|
| 规则层（必须常驻） | CLAUDE.md + 答疑 skill v2.3 + 答疑优先级 | **全量、固定** | ✅ 稳定前缀，命中省 90% 输入 |
| 知识层（按需） | 心得条目 / 真题 / 教材段 / 弱项 | **grep 工具实时拉命中片段** | ❌ 断点之后，不破前缀缓存 |
| 会话层 | 本轮对话 + 注入的历史摘要 | 动态 | ❌ 断点之后 |

- 上下文从 ~90K token → **~15-20K token**（规则层 ~25K 缓存命中后等效 ~2.5K + 检索片段 ~10K + 对话）。
- **修正第 1 轮"完整注入心得文件"的承诺**：心得 8 万字不再全量塞，改 grep 命中条目注入。
  理由：① 更省 ② 缓存才命中 ③ **grep 本就是 v2.3 的核心机制，全量注入反而是"伪 grep"**。
  ⚠️ 这是对第 1 轮决策的修改，需云确认（见文末待确认）。

### 杠杆 B：模型分层（单价 ↓，按任务用最便宜够用的模型）

| 任务 | 模型 | 理由 |
|---|---|---|
| 答疑（严谨辨析/案例） | **Opus** | 严谨性命脉，不可降级 |
| 背诵 L2 理解 / L3 应用 评测 | **Opus** | 开放式评分要法理判断，需 Opus |
| 背诵 L1 关键词比对 | **规则匹配（同义词表）→ 命中即 0 API** | 机械任务，零成本 |
| L1 规则匹配存疑时兜底 | **Haiku** | 极少触发，便宜 |
| 草稿/摘要生成（候选弱项/心得/历史摘要） | **Haiku / Sonnet** | 非严谨场景 |

### 杠杆 C：背诵题库预生成 + 批量评分（调用次数 ↓90%）

- **预生成**：一个考点的 L1/L2 检测题相对固定。**首次生成后存 Supabase 复用**，不每次实时生成。
  背诵检测的大头从"每天 N 次实时出题"变成"复用题库 + 偶尔刷新"。
- **批量评分**：一次背诵 session 做 10 道题，**攒成一批一次调用评分**，而非一道一调用。
- L1 走规则匹配 → 这部分**完全不花 API**。

### 成本模型（结构；单价待用 claude-api 参考核实后填实数）

```
日成本 ≈ 答疑轮数 × (缓存命中输入价 × ~2.5K + 输出价 × ~1.5K + 检索片段输入价 × ~10K)
        + 背诵 L2/L3 评测次数 × (缓存输入 + 批量输出)
        + L1 部分 = 0（规则匹配）
```
- 关键：**输出 token 永远全价**，是主要变量 → 控制 AI 回答啰嗦度（答疑精炼、评测给分简洁）。
- ⚠️ **实数单价必须先用 `claude-api` 技能拉 Opus/Haiku 最新价格**再算，本轮不臆造数字。
- 目标护栏：设 **API 日用量上限熔断**（P2-3），超额停服防失控。

## 2. 最科学：闭环数据飞轮（本系统真正的护城河）

三个系统不能是孤岛（P1-6）。科学的学习系统必须让数据**复利滚动**：

```
   ┌─────────────────────────────────────────────────────────┐
   │                                                         │
   ▼                                                         │
[答疑] 暴露错误/混淆                                          │
   │  → 写 Supabase 候选弱项（Haiku 起草）                     │
   │  → 云点 ✓ 确认                                           │
   ▼                                                         │
[弱项表] 错误频率 +1，带教材锚点                              │
   │                                                         │
   ▼                                                         │
[背诵调度] 公式 w2·弱项加权 → 对应考点自动插队/加权            │
   │                                                         │
   ▼                                                         │
[背诵检测] L1/L2/L3 ──通过──→ 回写"已强化项" ────────────────┘
   │                └─不通过─→ 错误频率 +1（继续滚）
   ▼
[雷达图] 按父考点聚合，反映真实掌握度（不是自我感觉）
```

### 实现关键：一张"链接层"表把三者钉在一起
考点、弱项、心得规则**共享锚点键**（教材页码 / 行号 / 法条号）：
- 每个弱项有「教材锚点」字段 → 映射到背诵考点
- 每个考点有「教材页码 + 关联心得编号 + 关联弱项编号」字段（第 2 轮已定）
- 答疑暴露的弱项，通过锚点找到对应考点，自动调高优先级

→ 这就是市面 APP 给不了的东西：**云的弱项/心得资产直接驱动背诵调度**。

### 科学性的其他落点
- **评分 rubric 化（P1-4）**：L1/L2/L3 每档定明确评分标准，杜绝 AI "自我安慰式给过"。
  - L1：关键词集合 + 同义词表，命中率 ≥80%（不是裸字符串匹配）
  - L2：必须复述出教材原文的法理逻辑要点（grep 比对，缺要点不给过）
  - L3：罪名/法律关系定性正确 + 说出"为什么适用此条而非彼条"（呼应苏格拉底式教学要求）
- **间隔重复优雅回落（P2-2）**：到期不"清零重来"，而是**退半档**（如 L3→L2 重测），避免雷达图天天暴跌打击信心。
- **冷点覆盖（P1-5）**：背诵题库 = 高频考点文件（已锚定真题）+ 教材全文结构化切分（补冷点）。
  用「按需懒切分」（P2-1）降低冷启动：调度到哪个冷点才切分+校正哪个，把 2-3h 集中校正摊到日常。

## 3. 最精：答疑忠实复刻 v2.3 的工程化

- **强制 grep 落地**：注册 `search_xinde / search_textbook / search_zhenti` 工具；
  system prompt 写死"回答前必须调用 ≥1 次 search"。
- **工具失败降级（P1-3）**：grep 返回空 / 网络失败时 → **强制标 ★ + 信心度封顶 50% + 显式提示"未取到锚点"**，
  绝不允许悄悄退化成"凭通说答"。这是 v2.3"无锚必标★"的工程兜底。
- **答疑优先级硬编码**：心得 → 真题 → 教材 → 法条 → 司法解释，工具调用顺序与返回拼装按此序。
- **跨会话记忆（P1-7）**：会话结束 Haiku 生成结构化摘要（已确认结论 / 未决疑点 / 暴露弱项），
  存 Supabase；新会话只注入**摘要 + 标记可信度**，旧的错误结论标注"待复核"防污染。

## 4. 同步与缓存（P0-2 / P0-3 的协调方案）

- **Supabase 存"易变结构化数据"**：答题日志、调度状态、SRS 间隔、对话历史、候选草稿队列。
- **GitHub markdown 存"内容真相源"**：弱项 / 心得 / 进度 / 教材。
- **单写者原则破冲突**：markdown 的落盘**只由 PC 端 Claude Code 做**（云确认草稿后落地）；
  手机端永不直接改 markdown，只往 Supabase 写草稿。→ 根除双写冲突。
- **手机端读最新**：答疑/背诵需要的最新心得/弱项，从 GitHub 读 PC 已 commit 的版本 +
  叠加 Supabase 未回写草稿展示。
- **缓存分层**：规则层（CLAUDE.md+skill+心得稳定区）做 cache breakpoint；
  PC 更新规则层后才重置缓存（低频），知识检索片段放断点后（不破缓存）。
