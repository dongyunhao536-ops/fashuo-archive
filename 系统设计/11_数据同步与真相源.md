# 数据同步与真相源设计（收口 架-1 / P0-2）

> 生成日期：2026-06-06（第 5 轮，最后一个 P0）
> 决策方式：云授权"你来决定"。本文给出**定稿设计**（非选项），解 `09` 架-1 / `01` P0-2 的三处隐患：
> ①"单一真相源"读取时变两库之并 ②跨库去重未设计 ③硬依赖 PC 每日运行。

---

## 0. 核心思路（一句话 + 生活类比）

> **让每一类数据只有一个"主人"，手机永不直接改正式档案，只往一个"待办筐"里塞便签；PC 当唯一的书记员，有空时把便签登记进正式档案。**

**类比**：法院书记员。律师（手机）随时往**收件筐**塞"立案申请便签"，但**只有书记员（PC）**能把案子登记进**正式卷宗（markdown）**。律师永远不直接动卷宗 → 不可能两个人同时改同一份卷宗打架。书记员三天没上班？申请便签在筐里等着，一份不丢，他回来一次性登记完。

---

## 1. 三类数据，各有唯一主人（解隐患①"两库之并"）

把"全部塞进 markdown"这个错误前提拆掉——**很多数据本就不该进 markdown**。

| 数据类 | 内容 | 唯一主人 | 手机 | PC |
|---|---|---|---|---|
| **A 内容资产** | 考点库内容、心得、教材、易混库、真题、CLAUDE.md | **markdown@GitHub** | **只读** | 读写（Claude Code） |
| **B 运行状态** | kp_state（间隔档/难度D/到期日/三档）、detection_log（检测流水） | **Supabase（永久，不回 markdown）** | 读写 | 只读 |
| **C 增量提案** | 手机产生的"候选弱项/候选心得/已强化" | **Supabase 待办筐（append-only）** | 只追加 | 读取并登记进 A |

**关键认知**：
- **B 运行状态根本不该进 markdown**——它是"运行时状态"不是"内容真相"。它只在 Supabase 一份，**没有第二份就没有同步问题**。（"两库之并"的混乱，一大半来自硬把状态往 markdown 塞。）
- 这样 **A 是纯内容真相（手机只读，永不写）、B 是纯状态（只在 Supabase）、C 是增量队列**——每个数据点只有一个权威家，不再"两库之并"。

---

## 2. 待办筐 + PC 唯一登记员（解隐患②跨库去重 + 双写冲突）

**弱项/心得的"双写"是唯一真冲突点**（既是内容想进 markdown、又被手机追加）。解法：

```
手机答疑/检测暴露卡点
   → 云在当场确认（在手机，即时）
   → 写一条"提案"进 Supabase 待办筐 proposals（append-only，只追加不改旧）
   ───────────────────（异步，不急）───────────────────
PC 开 Claude Code 时跑"登记"：
   → 读 proposals 里 status=pending 的
   → 按 科目+知识点 去重合并进 当前弱项.md
       命中 → 错误频率+1；未命中 → 新增一行
   → 标记该提案 consumed
   → git commit
```

- **手机永不编辑 当前弱项.md**，只往筐里追加 → **当前弱项.md 永远只有 PC 一个写者 → 不可能并发冲突**。
- **去重只发生在"登记"这一个地方**（PC、对着 markdown、一把钥匙 科目+知识点）→ 不存在"跨库去重竞态"。
- 心得同理：候选心得进 `proposals`(type=心得)，PC 登记时按"≥2 真题背书"等入选规则审核后并入心得文件。**统一一套模式**。

---

## 3. 手机完全不依赖 PC（解隐患③"PC 每日运行"硬依赖）

**问自己：手机日常学习，到底需要 PC 同步什么？答案是——几乎不需要。**

| 手机要读 | PC 关机几天会怎样 |
|---|---|
| **A 内容**（考点/心得/教材/真题） | PC 关机=云没在改内容→GitHub 上的内容本就没变→**手机读到的永远是最新**，零过时 |
| **C 弱项实时画面** | 手机读 = 正式档(markdown 镜像) ∪ 我自己的待办筐(pending 提案)。背诵调度用这个"并"做加权 → **手机永远看得到最新弱项**，不等 PC |
| **B 状态** | 本就在 Supabase，手机直接读写，**从来不需要 PC** |

> 这个"并"很干净，**不是冲突，是"已登记 + 我自己还没登记的"**——就像 git 的"已提交 + 工作区改动"。读的时候：提案的 科目+知识点 若已在 markdown，显示"+1 待登记"；真正的合并等 PC 登记。

**结论**：**手机日常学习对 PC 零依赖。** PC 同步从"必须每日跑否则崩"降级成"有空登记的后台杂务"。云一周不开 PC？待办筐攒着、一份不丢，下次开 PC 一次性登记完。**无发散、无丢失、无影响学习的过时。** 隐患③彻底消除。

---

## 4. "云确认"在手机（解隐患④）

- **确认在手机、即时**：答疑/检测当场暴露卡点 → 云在 flow 里一键"收下" → 进待办筐。体验顺、不打断。
- **正式登记延到 PC**：纯后台记账，云**不需要 PC 才能确认**。

---

## 5. 内容如何到后端供 grep（衔接 P0-3 缓存）

- PC push 内容到 GitHub → **GitHub Action 把内容镜像进 Supabase 只读内容表** content_mirror（+ 必要时触发 Vercel 重部署）。
- 后端（Vercel）对 content_mirror 做 grep（答疑/评分的 search 工具）→ **手机读内容也不依赖 PC 是否开机**（读 Supabase 镜像，永远是最后一次 push 的内容）。
- 规则层（CLAUDE.md+skill+心得稳定段）只在 PC push 内容时刷新 → **缓存前缀稳定**（呼应 `02` 缓存分层、解 P0-3）。

---

## 6. 同步触发时机

- **PC 开 Claude Code 时自动跑一次"登记"**（云坐到 PC 前就自动办，不用记着手动）。
- 额外提供手动命令"同步/登记"。
- **不时间敏感**：漏几天只是待办筐变长，登记时一次清完。

---

## 7. DB schema 草样（喂第 5 轮 schema 草案）

```
-- A 内容镜像（只读，GitHub Action 从 markdown 同步）
content_mirror(kind, path, chunk_no, content, updated_at)     -- 供后端 grep

-- B 运行状态（Supabase 唯一真相，不回 markdown）
kp_state(kp_id PK, 当前档, 难度D, 上次复习日, 下次到期日,
         L1状态, L2状态, L3状态, R_target, 优先级分, 累计复习, 累计错误, 已掌握)
detection_log(id PK, kp_id, 时间, 档位, 题目, 作答, AI评分, 是否通过,
              用时秒, 模型, grep命中行号, 信心度, ★标记)

-- C 增量提案（待办筐，append-only；PC 登记后 consumed）
proposals(id PK, type(弱项/心得/已强化), 科目, 知识点, 锚点, 来源(答疑/检测),
          payload, status(pending/consumed), created_at, consumed_at)
```
- 去重键：proposals 登记时按 (科目, 知识点) 对 当前弱项.md 去重。
- detection_log 是否滚动保留近 N 条 → `04` §9 待定项，留到 schema 定稿。

## 8. 这套设计兑现了什么

| 隐患（`09` 架-1） | 解法 |
|---|---|
| ①"单一真相源"读取变两库之并 | 三类数据各有唯一主人；状态不进 markdown；内容手机只读 |
| ②跨库去重未设计 | 去重只在 PC 登记一处发生，一把钥匙，无竞态 |
| ③硬依赖 PC 每日运行 | 手机日常零依赖 PC；同步降级为后台杂务，漏几天不坏事 |
| ④确认在手机还是 PC | 确认在手机即时，登记延到 PC 记账 |

**+ 诚实保留了"markdown=内容真相源"**：因为状态本不该进 markdown，增量是排队登记非直接写——这个口号现在站得住了。
