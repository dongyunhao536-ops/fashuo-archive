# 背诵系统 · 双视角深度评估（第 5 轮 review）

> 生成日期：2026-06-06（第 5 轮开篇）
> 方法：分别站在 **法硕教授**（教学法/应试有效性）与 **软件架构师**（可建性/可维护/成本/失败模式）两个角度，
> 对背诵系统做穿透式评估，给出动手开发前的结论。
> 依据：`00`(决策) `01`(漏洞) `02`(成本) `03`(引擎) `04`(数据结构) `05`(建库) `06`(清单) `08`(评估)。
> 原则：只挑真问题，不护短；两视角各自独立给结论，最后看是否收敛。

## 一、法硕教授视角

### 1.1 教学法内核：对路、科学（先肯定）
- **客观检测替代主观自评**（`03` 断裂点4）：法硕——尤其刑民案例、近年重理解——靠"以为会"必翻车。用"真答对"判掌握，方向对。
- **三档 L1/L2/L3 映射真实题型**：L1 再认↔单选、L2 理解↔简答、L3 应用↔案例。检测形态与考法对齐。
- **易混交错**（`03` §3.5）：直击云的头号敌人（概念混淆），而法硕选择题正是高密度辨析。教学法上是精准打击。
- **死线分级目标留存 + 阶段切换**（`03` §3.3）：契合经验帖"铺开→高频→冲刺"的阶段论。应试节奏正确。

### 1.2 四个必须正视的应试风险

**教-1 🔴 强项打偏靶心：系统优势区 ≠ 背诵主战场**
- 决策 2（`00`）：**法综（法理/宪/法制史）才是背诵主战场**（记忆量最大）；刑民只背规则层。
- 但整套系统目前是**在刑民上验证**的，而刑民恰恰是"理解应用"为主、最不靠死背的科。
- 更糟的倒挂（`05` 探测实据）：
  - 法综三科**连可结构化文本都没有**（只有 PDF），建库只能靠"高频考点文件"兜底。
  - 我们刚定的**导入校验**（拿教材文本 diff 揪错）——**法综无文本 → 无法自动校验**。
  - 即"记忆量最大、最依赖背诵、出错最致命（法制史细节）"的科，恰是**数据最薄、最难验证、最易把错的背进去**的科。
- **判语**：第一期做刑民是**练管线**，不是打主战场。真正的硬仗在法综，而那里地基最虚。最大风险＝把刑民背诵引擎做得很漂亮，然后卡在法综建库与校验上。

**教-2 🟡 冷点覆盖偏弱，正撞北大命门**
- 建库主源（高频考点文件）是**高频偏向**的（`01` P1-5）。补冷点靠"教材全文切分/PDF 提取 + 懒切分"，而法综冷点依赖尚未做的 PDF 提取。
- 北大 375+ 的边际分恰恰来自**不丢冷点**。一个结构上高频偏向的题库，可能系统性少喂正是拉开 350/375 差距的那些点。
- **判语**：对北大目标必须偏"全覆盖"，但这会显著抬高建库成本。"高频 vs 全覆盖"（P1-5）至今未决，且这是北大专属风险。

**教-3 🟡 检测题质量＝系统真实天花板（循环论证风险）**
- 客观检测的前提是"题靠谱"。L2/L3 题 `02` 说"预生成+复用+AI 生成"。但 **AI 出题 + AI 评分同源**，正是云在 CLAUDE.md 反复警惕的循环论证：可能"通过"只代表"答对了一道 AI 出的水题"，不等于"够北大"。
- 缓解是"L2/L3 优先用真题/模拟题"，但真题池有限（~24 卷），**盖不住 ~1400 考点**。绝大多数考点的 L2/L3 检测**仍得 AI 生成** → 循环风险在规模上真实存在。
- 尤其 **法综主观题（简答/论述，法综最高分题型）自动评分极难可靠**——这恰是系统最弱、却最该强的地方。
- **判语**：客观检测的优势真实，但天花板由检测题质量+评分纪律决定。系统最擅长客观题式检测，最不擅长法综主观题——而后者分值最重。

**教-4 🟡 考场手写 vs 系统点选/打字（`01` P2-5）**
- 法综主观题靠手写默写大段，得分含速度+肌肉记忆。L1 点选/打字练不到"手写产出"。北大主观题占比高 → 真实迁移缺口。

### 1.3 教授结论
> 教学法内核科学、对路；但**系统的强项区（刑民/客观题/理解应用）不是法硕背诵的靶心，靶心（法综海量记忆、主观题、冷点边际分）正是数据最薄、检测最难自动化、最易出错且最难验证的区**。这是"强项打偏靶心"的结构性错配。可以做，但要清醒：刑民是练兵，法综才是决战，且别让高频偏向稀释了北大要的冷点。

## 二、软件架构师视角

### 2.1 架构骨架：扎实（先肯定）
- **三层数据划分干净**（`04`）：定义层(markdown/git，可 grep 可版本) / 状态层(Supabase) / 记录层(Supabase)，考点ID 串联，单写者破冲突。把"低频内容"与"高频状态"分离，是正解。
- **markdown 真相源 + 参数外置**：调度权重/间隔档/rubric/模型分层全可配置——可扩展性确实落地了（呼应云"越用越懂、必然迭代"的硬要求）。
- **成本三杠杆**（`02`）：规则常驻+grep 检索 / 模型分层 / 题库预生成+批量评分——方向对，"grep 本身即省钱钥匙"是真洞察。
- **锚点规范化层**（`04` §6）：让"考点⇄弱项⇄心得⇄易混⇄真题"自动关联的链接键，是飞轮的数据地基。设计漂亮。

### 2.2 两个 P0 级阻塞 + 三个 P1

**架-1 🔴 同步模型内部不自洽，且硬依赖 PC 运维（P0-2 未真正解决）**
- `02` §4"单写者：markdown 只 PC 落盘，手机只写 Supabase 草稿"破了双写冲突——但引入新洞：
  - 手机答疑沉淀的弱项，"云确认"在手机点（写 Supabase），最长 24h 后才进 markdown。期间**背诵调度读弱项加权，必须读 markdown ∪ Supabase 待回写**——即"单一真相源"在读取时其实是 fiction，真相永远是两库之并。
  - **跨库去重未设计**：弱项去重键"科目+知识点"必须同时跨 markdown 和 Supabase 去重，否则同一弱项两库各记一次。
  - **PC 不开机的降级未定义**：云在职，PC 时间正是最稀缺的。若三天没开 PC → markdown 心得/弱项变陈旧 → 答疑缓存读到旧结论。`01` P0-2 早警告"单用户系统最容易死在运维纪律上"。
- **判语**：同步比 v1 干净，但仍有 (a) 读时一致性破"单一真相" (b) 跨库去重未定 (c) 对 PC 每日运行的硬依赖无降级。**对在职用户这是头号架构脆弱点。**

**架-2 🔴 成本未落地实数，可行性未验证（P1-1）**
- `02` 明确把单价留作"待 claude-api 拉取"；`01` P1-1 警告实际可能是估算的 2-4 倍，且**背诵检测（L2/L3 Opus 评分+多轮 grep）可能比答疑更烧钱**。
- "月 40-70 元"（`00`）未经验证。一个跑 6+ 个月的日用系统，若真实是 150-250/月，决策基础就变了。
- **判语**：没有真实成本模型不能开工。这是"动手"的 P0 阻塞。必须先拉 Opus/Haiku 实价 × 估背诵检测调用量。

**架-3 🟡 "题库复用省钱" 与 "轮换防背题" 直接矛盾（跨档冲突）**
- `02` 杠杆C：题库预生成、存 Supabase 复用、不每次实时生成（省钱）。
- 本会话刚定（`00`）：检测题不固定重复、同档换变体防背题。
- 两者反向拉扯：复用省钱 vs 轮换需要多样性（多生成=更贵）。合理解（每考点预生成一个变体池、池内轮换）可行，但：①~1400 考点 × 多变体 × 三档的**预生成是笔大的前置成本**；②L3 案例换情境很难全预生成。
- **判语**：真实矛盾，需明确分层策略（L1 规则生成变体≈免费；L2 预生成 N 个池；L3 情境受控按需生成），否则两份文档互相打架。

**架-4 🟡 先做"更复杂的背诵"+在职 = 备考挤占最大化（P1-8）**
- `01` P1-8 本就建议：P0 砍到只剩答疑（1 周可用），背诵挪暑期。背诵是两核中**更复杂**的（建库管线+调度引擎+三档检测+评分+飞轮+同步）。
- 在职、晚上开发，背诵单独落地现实工时估 4-6 周晚间，期间备考被挤。**开发时间从备考扣——一个备考工具反而吃备考时间，是其自身最大的 ROI 悖论。**

**架-5 🟡 法综 PDF→文本是未解工程债（主战场建库 blocker）**
- `05`：法综只有 PDF。高频文件能 bootstrap 高频，但全覆盖/冷点/导入校验都依赖 PDF 干净提取——质量与工时未定义。这块不解决，教-1 的主战场就一直悬。

### 2.3 架构师结论
> 数据架构（三层 + 锚点规范化 + 参数外置）是全案最扎实处，可扩展性达标。但**两个 P0（同步读时一致性+PC 运维硬依赖、成本未落实数）会真正阻塞"动手"**，外加题库"复用 vs 轮换"内部矛盾、法综 PDF 工程债、以及"先做更复杂的背诵+在职"的备考挤占。**骨架可建，但不是现在直接开工的状态**，这几条 P0/P1 必须先收口。

## 三、综合结论（两视角收敛于同一句）

> **背诵系统设计水平高，但它的"强"集中在考试最不需要的地方（刑民/客观题/理解应用），它的"弱链"（法综数据、检测题质量、同步运维、成本）全部聚集在考试最需要的地方（法综海量记忆）或"动手"被卡住的地方。**

两个视角独立分析，撞到同一结论：**不建议现在直接全量开工背诵系统。** 先收口下列硬事。

### 3.1 开工前必须收口（P0）
1. **成本落地实数**：claude-api 拉 Opus/Haiku 实价 × 背诵检测调用量估算 → 验证 6 个月可行 + 设日熔断。【架-2】
2. **同步模型补完**：读时去重（markdown∪Supabase）、PC 不开机降级、"云确认"在手机还是 PC。【架-1】
3. **题库"复用 vs 轮换"策略 + 法综验证缺口**：法综到底有没有靠谱的考试分析 ANKI 卡组？没有的话主战场内容从哪来、怎么校验。【架-3 + 教-1】

### 3.2 范围与里程碑建议
4. **第一期建库＝刑法**（管线零障碍，`05` 已验证）——但定性为**练管线**，非主战场。
5. **尽早做法综小样验证**：拿一章法制史试走"导入→检测"全流程，**用最小样本探明法综能不能跑通**，别等刑民全做完才发现法综塌方。这是决定整个背诵系统 ROI 的关键里程碑。
6. **守北大命门**：冷点覆盖 + 法综主观题手写——两处 375+ 边际分来源现设计都偏弱，要么补强，要么明确"这部分仍靠 PC/纸笔"。

### 3.3 一个值得重新审视的顺序问题
- `01` P1-8 原建议"先答疑（1 周可用）、背诵挪暑期"；本会话云选了背诵优先。
- 值得重申这个 trade-off：**答疑更轻、更快出价值，且背诵的检测评分纪律本就是从答疑 v2.3 派生的**——先做答疑，等于为背诵检测先把"评分引擎 + grep 工具链"趟通，背诵再做时复用现成。
- **不是否定云的选择，而是建议：等 3.1 第 1 条（成本实数）出来后，再最终拍"背诵先 vs 答疑先"。** 若背诵检测成本确认是吞金兽，答疑先做的理由会更强。
