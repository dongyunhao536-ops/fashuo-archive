## 🔴 P0：动摇地基的根本性质疑

### P0-1 「为什么不用现成方案」这个根本假设 —— ✅ 已决策（2026-06-05）
- **云的决策**：**不考虑 Tailscale/远程方案**。背诵与答疑都是核心，都自建，要做到"最精、最科学、成本可控"。
- **结论**：自建双核路线确定。重心转向「如何在自建前提下同时满足 精/科学/省」→ 见 `02_精科学与成本控制设计.md`。
- ~~原顾虑（远程方案可替代答疑）~~：云已明确否决，存档备查。

### P0-2 双向同步的真相源冲突 + 24h 延迟
- **问题**：声称"markdown 是单一真相源"，但手机只写 Supabase、PC 每日脚本回写 markdown。
  - 手机白天沉淀的弱项，最长 24h 后才进 markdown；当晚回家用 PC 复盘时 CLAUDE.md/弱项仍是旧的 → 重复劳动甚至冲突。
  - PC 端 Claude Code 也在改同一批 markdown（弱项/心得），与 Supabase 待回写草稿**双写同一文件**，每日脚本如何 merge？没有冲突解决策略。
  - 同步脚本手动还是定时？漏跑一天，雪球越滚越大。**单用户系统最容易死在"运维纪律"上**。
- **待答**：要不要改成"手机端也能实时拉/写 GitHub"或"近实时同步"，把延迟从 24h 压到分钟级？冲突解决规则怎么定？

### P0-3 「实时拉最新 markdown」与「Prompt Caching 命中」直接矛盾
- **问题**：答疑要注入最新 CLAUDE.md+skill+心得；但 Prompt Caching 要求前缀**逐字节一致**才命中。
  - 若每次实时拉最新内容 → 前缀天天变 → 缓存命中率暴跌 → 第 1 轮"缓存后 0.05 元/轮"的成本估算**直接作废**。
  - 若构建时打包进去 → PC 改 markdown 必须重新部署 Vercel 才生效，对"每天更新心得/弱项"的系统是致命的。
- **改进方向（同时解 P0-3 与 P1-1）**：**缓存分层**——把几乎不变的（CLAUDE.md+skill+心得稳定部分）做成稳定缓存前缀；把频繁变的（最新弱项、今日上下文）放在缓存断点之后。既保命中率又保新鲜度。

## 🟡 P1：开发前必须定的质量/成本问题

### P1-1 成本估算过于乐观（可能是实际的 2-4 倍）
- 漏算 1：**背诵系统的检测调用量**第 1 轮完全没进成本模型。每天大量 L1/L2/L3 出题+评分，每道都是一次（甚至多轮 tool use）Claude 调用——**背诵可能比答疑更烧钱**（高频、小颗粒、多轮 grep 往返）。
- 漏算 2：**输出 token 永远按全价**，缓存只省输入。一轮答疑输出 1-2K token，Opus 单轮就不止 0.05 元。
- 漏算 3：**grep tool use 多轮往返**会把工具结果反复塞回上下文，单次"回答"总 token 翻几倍。
- **待办**：重做成本模型，把背诵检测调用量算进去。

### P1-2 全程 Opus 是浪费，错失最大省钱点
- **问题**：L1 关键词命中判断这种机械任务用 Opus 是杀鸡用牛刀。
- **改进**：**分层用模型**——L1 评分用 Haiku/规则匹配，L2/L3 评测与答疑用 Opus。第 1 轮"必须 Opus"是针对"严谨答疑"，不该一刀切套到背诵检测。

### P1-3 强制 grep 的「工具失败降级」未定义
- **问题**：system prompt 写死"回答前必须调 search 工具"。但 GitHub 拉取失败/grep 没命中/网络抖动时，工具返回空——Claude 是硬编答案还是拒答？v2.3"无锚必标★"在 Web 端怎么落地？
- **风险**：线上一抖动就退化成"凭通说答"——正是 v2.3 要根治的病。
- **待办**：定义工具失败时的降级行为（建议：返回空时强制标 ★ + 信心度封顶 + 提示"未取到锚点"）。

### P1-4 背诵 L1/L2/L3 评分的假阳/假阴
- **L1**：法律术语近义（"定罪量刑通则" vs "总则性质"）机械字符串匹配会误判 → 需语义判断，但又回到烧 Opus。
- **L2**："grep 教材比对法理是否说出"如何自动判断？最容易**AI 自我安慰式打分**（答得含糊也给过）→ 必须 rubric 化，否则 L2 形同虚设。
- **L3**：同一案例多个合理答案，评分标准谁定？
- **待办**：每档定明确 rubric；L1 考虑"关键词集合+同义词表"而非裸字符串。

### P1-5 背诵考点的「冷点覆盖盲区」
- **问题**：高频考点文件只覆盖高频。以它为初始题库 → 低频但仍可能考的冷点被系统性遗漏。
- **矛盾**：决策 3 特意修了"冷考点不能永远排不上"，但**数据源根本没收录冷点**的话，公式修正是空的。
- **冲突点**：北大 375+ 靠的恰是"不丢冷点的边际分"。
- **待答**：背诵是"高频优先"还是"全覆盖"？若全覆盖，初始题库必须回到教材全文切分（工作量/校正成本大增）。

### P1-6 答疑 ⇄ 背诵 ⇄ 弱项 没有闭环（目前是三个孤岛）
- **理想数据飞轮**：答疑暴露弱项 → 自动给对应背诵考点加权/插队 → 背诵检测验证是否真掌握 → 回写"已强化项"。
- **现状**：背诵检测发现的弱点是否回写 `当前弱项.md`？答疑沉淀的弱项是否调高对应考点优先级？文档里没设计。
- **待办**：把三者设计成一个闭环，这是本系统相对市面 APP 的真正护城河。

### P1-7 跨会话记忆「手动管理」是答疑质量的隐形杀手
- **问题**：风险清单一句话带过"Supabase 存历史摘要，新对话注入"。但摘要由谁生成、何时、注入多少、会不会把**错误的旧结论**带进新会话——全没定。
- **对比**：这正是 Claude Code memory 的核心能力，手动复刻极易做坏。
- **待办**：定义摘要生成时机/长度/可信度标记。

### P1-8 3 周 P0 在「在职+备考」双挤占下不现实
- 风险清单自己也承认"范围扩张风险"。开发时间从备考扣。
- **更现实的路线**：P0 砍到只剩答疑（若 P0-1 远程方案不成立的话，1 周能用）；背诵系统延到背诵需求最旺的**暑期**再做。
- **待答**：是否接受"开发期备考停摆"，还是把背诵系统挪到 7-8 月？

## 🟢 P2：优化项（可迭代，不阻塞开发）

### P2-1 冷启动改「按需懒切分」
- 不必一次切 2500 考点全校正。**调度到哪个考点才切分+校正哪个**，把 2-3 小时集中校正摊薄到日常，降低冷启动门槛。

### P2-2 「已掌握→回落」的信心体验
- 间隔重复到期回落重测，对时间紧的云可能产生"永远背不完、进度倒退"的挫败感。
- **改进**：UX 区分"新学考点"vs"维持性复习"，雷达图避免天天掉分打击信心。参考经验帖的复习节奏方法论。

### P2-3 鉴权与烧钱熔断
- PWA 暴露公网 + 密码登录，被撞库后有人薅 API 烧钱。
- **改进**：至少加 IP allowlist 或更强鉴权 + **API 日用量上限熔断**。

### P2-4 v2.3「评测模式」在 Web 端的承载
- v2.3 大量机制针对"做一整卷"（候选心得隔离、笔误嫌疑暂停）。Web 端以单题问答为主。
- **待定**：Web 端只做单题答疑、整卷评测仍回 PC？还是 Web 也支持整卷模式？需明确分工。

### P2-5 「考试是纸笔」的对齐
- 背诵系统训练"对手机打字/选关键词"，实际考试是**手写**主观题。L1 打字默写 vs 考场手写，肌肉记忆和速度不同。
- **改进**：主观题模板背诵可考虑"手写拍照"而非打字（非紧急，记一笔）。
