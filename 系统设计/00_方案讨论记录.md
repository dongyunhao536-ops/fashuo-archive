## 已对齐的核心定位

### 系统目标
基于 `D:\fashuo` 现有档案体系，构建一个**为云一人定制**的 iOS 端备考辅助系统，解决"必须在家用 PC 才能学习"的痛点。

### 差异化护城河
不是做又一个法硕题库 APP，而是把云已经积累的**判断倾向手册 + 弱项档案 + 心得文件**在手机上激活——这些资产是市面 APP 给不了的。

### 不做什么（明确砍掉）
- 通用刷题题库
- 错题闪卡 SRS（错题从答疑中自然沉淀，不单独刷）
- 推送通知（仅 APP 内提醒）
- PC 端管理界面（PC 端继续用 Claude Code）

## 背诵系统设计（核心 1）

> 第 2 轮（2026-06-05）专题深化。以下为已锁定的详细设计。

### 现有档案的三个数据底座（第 2 轮勘查结论）
1. **教材原文自带切分锚点**：`教材/考试分析_文本.txt`（9869 行）每个考点标题已带页码 + 历年真题年份与题型，
   例：`一、罪刑法定原则的基本内容与体现（P11）（2019 法/非法单，2023 法/非法多）`。
   → 考点切分可先用结构规则抽一遍，不必纯靠 AI 猜。
2. **高频考点文件 = 半成品考点表**：`真题分析/0X_高频考点.md` 的每个 bullet 即"可独立成题"的考点，
   且已锚定真题题号，例：`具法=定罪量刑通则（2024-48D）`。→ 直接复用为背诵系统初始题库。
3. **弱项档案自带量化字段**：`薄弱知识点/当前弱项.md` 每行有"错误频率""最近出错日期""教材锚点"，
   → 直接喂调度公式当"弱项权重"，无需另建。

### 决策 1：考点切分 = 中粒度 + 父子两层
- **子考点**（调度最小单元）= 高频考点文件的 bullet 粒度（如"八议入律""五服制罪"各一个），
  与"一道选择题考一个点"的真题命题粒度对齐。预估 1500-2500 个。
- **父考点**（统计/雷达图聚合）= 节/标题（如"罪刑法定原则"）。L1/L2/L3 进度向上汇总到父考点再喂雷达图。

### 决策 2：覆盖范围 = 法综主战场 + 刑民只背"规则层"
- **法综（法理/宪/法制史）= 记忆主导**：背诵系统主战场。多数考点封顶 L1（如"铸刑书=子产"无 L3 场景），少数到 L2。
- **刑民 = 理解应用主导**：背诵系统**只背规则层**——罪名构成要件、民法制度规则、两份心得文件的 35 条规则；
  这些点封顶 L3。刑民的"理解"靠答疑系统，不靠背。
- 每个考点打标签 `最高档位 = L1 / L2 / L3`，不是每个点都走完三档。

### 决策 3：调度公式 = 加权和 × 遗忘门控（修正原纯乘法）
原 `优先级 = 紧迫度 × 弱项 × 遗忘 × 真题频率` 有 bug：任一项为 0 则整体归零，真题频率=0 的冷考点永远排不上。改为：
```
优先级 = (w1·真题频率 + w2·弱项加权 + w3·科目权重) × 遗忘紧迫度
```
- 真题频率：高频考点文件的 高/中/低 + ⭐ → 映射 3/2/1
- 弱项加权：取 `当前弱项.md` 的"错误频率"
- 科目权重：当前阶段法综 > 刑民记忆点（随阶段调）
- 遗忘紧迫度：距上次复习天数 ÷ 当前记忆档间隔，>1 才到期
- **"距考试紧迫度"不当乘子，改当全局模式开关**（呼应经验帖阶段论）：
  现在（剩 6 月）按知识体系铺开 → 9 月起切"高频+弱项优先" → 考前 2 周切"纯弱项+主观题模板"。

### 决策 4：间隔重复 = 简化间隔档（保留，非第 1 轮砍掉的"错题闪卡 SRS"）
- 固定档位 **1 / 3 / 7 / 15 / 30 天**；记忆档每升一级拉长间隔；到期回落重测。
- 注：第 1 轮砍的是"错题闪卡 SRS"（错题从答疑自然沉淀，不单独刷）；背诵考点的间隔重复是背诵系统的核心机制，保留。

### 决策 5：每日任务 = 算法跑分 + AI 草稿 + 云可改（与答疑"半自动"一致）
- 每日 6 点算法重算优先级 → AI 生成今日清单草稿 → 云可手动增删 / 调顺序。

### 背诵程度三档量化（检测题型与评分）
| 等级 | 含义 | 题型 | 评分标准 | 来源 |
|------|------|------|---------|------|
| L1 记忆 | 能背出关键词 | 填空/关键词回忆 | 关键词命中 ≥80% | 法综 AI 生成 |
| L2 理解 | 能解释法理 | "为什么/区别" | grep 教材原文比对法理是否说出 | AI 生成 |
| L3 应用 | 案例中识别 | 迷你案例定性 | 罪名/法律关系是否正确 | 刑民挑真题/模拟题 |

- 晋级：L1 过才测 L2，三档全过=已掌握；已掌握的点**过遗忘间隔会回落，需重测**。
- 反喂雷达图（按父考点聚合）。

### 主动检测
- 仅在 APP 内提醒（云已确认不要 push 通知）
- Claude 评分严格遵守 v2.3 "必须 grep 教材原文"
- 检测题来源：法综 → AI 实时生成；刑民 → 挑选真题/模拟题

## 答疑系统设计（核心 2）

### 必须严格复刻 PC 端
- 完整注入 CLAUDE.md + 答疑 skill v2.3 + 两份心得文件
- 严格按答疑优先级：心得 → 真题 → 教材 → 法条 → 司法解释

### 核心交互形态 = 引导式（苏格拉底），不直接给答案（决策 2026-06-05）
> 云的诉求：主动思考能力弱，要靠答疑**练这个能力**。游戏化/模拟法庭作废（花哨低效）。
> 背书：① CLAUDE.md 教学要求"苏格拉底式提问引导"；② `03` 测试效应——自己推出的结论远比被喂的牢。

**关键前提**：引导 ≠ 装不知道打哑谜。**我内部必先按 v2.3 把答案/法条/争点查清（grep 教材/心得/真题），心里有数才引导**，否则是瞎带。

**五步流程**（第 0 步内部，云只见第 1 步起）：
0. 〔内部〕v2.3 预检：查清答案、法条、争点
1. **拆题不给答**：帮云标题干信号词，但让云说"这题在问什么、几个主体、核心争点是啥"
2. **抛一个引导问题**：顺着争点问一层，逼云调取知识
3. **云答 → 不判对错，追问**：走对→往深追一层（法理为何）；混淆→**用反例反问**让云自己撞见矛盾
4. **云推出结论后才收口**：补全 + v2.3 三段证据卡（法条/教材页码/真题题号/心得编号）+ 完整法理
5. **沉淀**：过程暴露的卡点 → 候选弱项 → 云确认 → 写弱项数据源（飞轮闭合）

**形态 = 题型分流（路 B，2026-06-05 定）**：
- **选择题**：轻引导（给 1-2 个提示问题，云仍自己选，不来回拉扯）
- **案例 / 简答题**：重引导（逐层拆解——这是"主动思考"最该练、最值得花时间处）

**两个刹车（在职防折磨）**：
- **逃生舱**：云随时说"直接讲" → 立即跳过引导给完整答案（时间紧时效率优先，不强按头）
- **不死磕**：同一步卡 2-3 轮推不动 → 主动降档给提示 → 再不行直接揭答案（防挫败反噬）

**协同红利**：引导式天然暴露云卡在哪一步 → 弱项来源②抓得更准（每步推导都是诊断信号）。
**成本提示**：多轮引导比"一问一答"耗 token，但 prompt caching 命中后每轮约 0.05 元，可接受。

### 弱项（错题）数据源 = 双来源 + 单一真相源（决策 2026-06-05）
> APP 不含刷题，弱项不靠刷出来，两个口子灌入：

| 来源 | 在哪产生 | 写入路径 | 触发 |
|---|---|---|---|
| **① PC 手动录入** | 云做完真题/模拟题自己录 | Claude Code 直接写 `薄弱知识点/当前弱项.md` | 云 |
| **② 答疑沉淀** | 答疑暴露的混淆/答错 | APP 答疑 → 候选弱项草稿 → **云确认** → Supabase → 每日同步回写 markdown | 半自动 |

- **真相源 = markdown**：`当前弱项.md` 唯一权威；手机端只写 Supabase、每日脚本回写，**APP 绝不直接改 markdown**（防双写冲突，`01` P0-2）。
- **去重键 = 科目+知识点**：来源①②撞同一点 → **错误频率 +1**，不新增行。
- **每条标 `来源:录入/答疑` 字段**，便于回看是真做错还是聊出来的。
- **冷启动从 0**：现有 `当前弱项.md` 是 Claude 自测演练数据，不迁；真实弱项从这两口子重长。

### 技术难点与解决方案

**难点 1：v2.3 强制 grep**
- 解决：Claude Tool Use 注册三个工具
  - `search_textbook(关键词)` → grep 教材
  - `search_xinde(关键词)` → grep 心得
  - `search_zhenti(年份, 题号)` → 读取真题
- system prompt 写死"回答前必须调用至少一次 search 工具"

**难点 2：上下文 8-10 万 token**
- 解决：Claude Prompt Caching（5 分钟 TTL，命中后输入成本降 90%）
- 预估成本：首次 0.5 元，缓存命中后续每轮 0.05 元
- 一天聊 1 小时约 1-2 元，月成本 30-50 元

**难点 3：自动沉淀**
- 答疑结束触发词："懂了 / 下一题 / 下课"
- AI 生成草稿（候选弱项 / 候选心得 / 候选背诵任务）
- **半自动**：草稿进"待确认"队列，云点 ✓ 收下或 ✗ 丢弃
- 收下的写 Supabase → 每日同步脚本回写 markdown

**难点 4：iPad 长案例题**
- 题干折叠/展开
- 信号词自动高亮（呼应 v2.3 信号词标注）
- 支持 Apple Pencil 批注

### 数据写回策略（2026-06-06 升级为"待办筐+PC 登记员"，详见 `11`）
- **三类数据各有唯一主人**：A 内容资产=markdown@GitHub（手机只读）；B 运行状态 kp_state/detection_log=Supabase 永久（**不回 markdown**）；C 增量提案=Supabase 待办筐（append-only）
- **手机永不编辑正式 markdown**，只往待办筐追加提案（答疑/检测沉淀，云在手机即时确认）
- **PC 是唯一登记员**：开 Claude Code 时读待办筐 → 按 科目+知识点 去重并入 当前弱项.md/心得 → git commit（去重只此一处，无竞态）
- **手机日常零依赖 PC**：内容读 Supabase 镜像（GitHub Action 同步）、弱项读"markdown 镜像∪我的 pending 提案"、状态本就在 Supabase。PC 漏开几天=待办筐变长，无丢失无发散
- markdown 仍是**内容**真相源（状态本不该进 markdown，增量排队登记非直接写 → 口号成立）

## 已知 tradeoff / 风险

1. **Web 端 AI ≠ Claude Code 但≈ 95% 等价**：
   - 底模型同为 Claude Opus 4.7/4.8，差的只是 Claude Code 这层"壳"（自动加载 CLAUDE.md / skill / memory / 内置工具）
   - 复刻方式：system prompt 硬塞 CLAUDE.md + 答疑 skill + 心得文件 + Tool Use 注册 search_textbook/search_xinde/search_zhenti 等工具
   - 数据完全可放云端（GitHub + Supabase），不影响行为对齐
   - 唯一差距：跨会话记忆需手动管理（Supabase 存历史摘要，新对话注入）
2. **Claude Pro 账号无法代替 API**：Pro 与 API 是两条独立产品线，Pro 不能在自建网站做程序化调用；必须单独充 API 额度。反代 Pro 违反 TOS，不考虑。
2. **模型选型**：必须 Opus 4.7/4.8，Sonnet 在严谨答疑上稳定性差。
3. **开发时间从备考时间扣**：3 周开发期间云的备考时间会被挤占，需做时间预算。
4. **背诵数据冷启动**：AI 切分考点必然有错漏，云需花 2-3 小时人工校正才能用。
5. **MVP 范围扩张风险**：仪表盘 + 答疑 + 背诵 + 自动沉淀 + 同步脚本，单人 3 周偏紧。
