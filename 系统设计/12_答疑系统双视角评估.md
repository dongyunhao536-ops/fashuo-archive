## 一、法硕教授视角

### 1.1 教学法内核：对症（先肯定）
- **引导式（苏格拉底）直击云的真痛点**：云自陈"主动思考能力弱"，而引导式靠**生成效应/测试效应**——自己推出的结论远比被喂的牢（`03` 已论证）。这是 CLAUDE.md 明文教学要求，方向完全正确。
- **题型分流（路 B）是聪明的取舍**：选择题轻引导（不来回拉扯）、案例/简答重引导（逐层拆解）。把宝贵的"慢功夫"押在**主观题**这个最该练主动思考、也最值分的题型上，分流逻辑应试。
- **证据卡可追溯**（法条/页码/真题题号/心得编号）：收口阶段给完整证据链，教学严谨、可复查，符合 v2.3 纪律。
- **弱项从答疑自然沉淀**：呼应 CLAUDE.md"错题唯一归口=当前弱项"，不另设错题本，干净。

### 1.2 四个必须正视的应试风险

**教-1 🔴 引导式有"适用边界"，强行全用会空转**
- 苏格拉底式的**隐含前提**：云得有"可被追问的知识基础"。引导只在"**云有基础但没想透/混淆**"时高效——这正是刑民理解题、易混概念辨析的场景。
- 但对**完全陌生的法综记忆点**（背诵主战场：法制史细节、宪法条文数字），引导=对着空气提问，云答不出 → 卡 2-3 轮降档 → 最后还是揭答案。**绕一圈回到讲授，纯浪费在职时间**。
- 即：引导式的强项区是"理解类"，恰好**不覆盖**法综海量记忆——这与 `09` 背诵系统"强项偏离法综靶心"是**同构的结构性错配**。区别在于答疑本就主打理解，所以这个错配可接受，但**前提是分型清楚**：记忆类陌生点该直接讲，别硬套苏格拉底。
- **判语**：引导式必须配一张"**分型表**"——哪类题引导（理解/案例/易混）、哪类题直答（陌生记忆点、纯法条定位）。否则对在职的云，强行引导记忆类题是 ROI 倒挂。

**教-2 🔴 引导式把 v2.3 严谨预检藏进"内部"，反而更难监督"伪执行"**
- 五步流程第 0 步"内部 v2.3 预检（查清答案/法条/争点）"是引导质量的命根——**心里没数就引导=瞎带**，会把云往错误争点上带，比直接答错更隐蔽。
- 但在 PC 端 Claude Code 里，预检痕迹（grep 关键词 + 命中行号）是**显示出来、可审计**的（机制⑨的全部价值）。一旦搬进 APP 且"引导时折叠展示"，内部预检做没做、做得对不对，**云看不见也验不了**。
- 这正是 `08` A-2 R3 + 机制⑨警惕的"伪执行"在引导式下的**加剧版**：因为反正不直接展示答案，模型更容易在第 0 步走过场。
- **判语**：引导式不是减轻了 v2.3 纪律，是**更依赖它、却更难观察它**。必须保留内部预检的可审计痕迹（哪怕折叠，展开能看到 grep 行号），否则引导质量不可控。

**教-3 🟡 OCR 题干一字之差直接颠覆答案**
- 法律题干对文字极度敏感："善意/恶意""已交付/未交付""3 万/30 万"——OCR 错一个字，整道题的法律关系就变，引导从第一步就建立在错题干上。
- v2.3 机制⑫"笔误嫌疑暂停求证"是为**真题里罕见的笔误**设的；而 OCR 错误**高频得多**，⑫ 兜不住量。
- **判语**：拍照导题必须强制"**OCR 后云确认/修正题干**"一步，且引导开始前 AI 应主动复述题干关键事实（机制②本就要求）让云核对。不能 OCR 完直接进引导。

**教-4 🟡 答疑读的心得/弱项若混入自测演练数据，引导争点会被污染**
- 答疑要"严格复刻 PC 端"、读两份心得文件作答。但心得正文里可能混有标注"0.5 次/自测"的条目（CLAUDE.md 防循环论证规则）。
- 引导式靠"心得→真题→教材"链条找争点。若争点锚在**自测产物**上，等于用演练数据带真实学习——与"冷启动从 0、不迁自评学情"的决策自相矛盾。
- **判语**：答疑注入的心得/弱项必须只取**真题背书**部分，自测候选条目（待观察区）不进引导的"已知争点库"。这条要在 system prompt 注入策略里写死。

### 1.3 教授结论
> 引导式对症云"主动思考弱"的核心病，题型分流也聪明；但它有三道暗礁：**(1) 适用边界不清——记忆类陌生点强行引导=空转浪费在职时间，必须分型；(2) 它把 v2.3 严谨预检藏进内部，反而更难监督"瞎带"；(3) OCR 错字与自评数据污染会从源头带偏引导**。引导式值得做，但"对谁、用多重、内部查没查、题干对不对、争点干不干净"这四件事不定清，它就是把简单的一问一答换成一个更难验证质量的黑箱。

## 二、软件架构师视角

### 2.1 工程骨架：复用扎实（先肯定）
- **grep 硬约束工程化落地**：Tool Use 注册 `search_textbook/search_xinde/search_zhenti`，system prompt 强制"回答前至少调一次 search"——把机制⑨从"自律"变成"可强制"，方向对。
- **同步模型直接复用背诵**（`11` 待办筐 + PC 登记员）：答疑沉淀走同一条"手机写 Supabase 提案筐 → PC 去重登记"管线，不另起炉灶，省工。
- **Prompt caching 降成本**：稳定前缀（CLAUDE.md+skill+心得稳定段）缓存、易变段放断点后，命中后输入成本降 90%，合理。
- **比背诵简单**：答疑无建库管线、无三档检测、无调度引擎——单从"数据流"看，答疑确实是更轻、更快出价值的核（`01` P1-8 的判断成立）。

### 2.2 两个被低估的复杂度 + 三个 P1

**架-1 🔴 "引导式多轮"把最轻的核变成有状态对话机，复杂度+成本双被低估**
- 答疑本可以是**无状态一问一答**（最简）。但引导式是**有状态多轮**：要追踪"现在第几步、这一步卡了几轮、该不该降档、该不该收口、逃生舱有没有被触发"。
- "卡 2-3 轮降档""同一步推不动揭答案"需要**对话状态判断**。靠 system prompt 硬塞规则让 Opus 自己数"卡了几轮"——**LLM 自己计数/管状态不可靠**（容易该降不降、或过早放弃）。要么显式建状态机（工程量上去），要么靠模型自觉（行为漂移）。这是被"答疑很简单"掩盖的真实工程债。
- **成本侧**：重引导一道案例题 5-8 轮，每轮都带 8-10 万 token 上下文。即便缓存命中（输入降 90%），**输出 token 全价 × 轮数**累加。`00` 自己承认"多轮比一问一答耗 token"，`10` 成本模型里答疑 ¥50-100/月**是否按"重引导多轮"估的**需核——若按一问一答估，重引导会击穿。
- **判语**：引导式不是答疑的免费交互糖，它同时抬高**实现复杂度（状态机）**和**单题成本（多轮全价输出）**。这两项都在"答疑很轻"的直觉下被低估了。

**架-2 🔴 内部预检不可观测 → 引导质量无法验证（与教-2 同根，工程侧）**
- PC 端 grep 痕迹是显示的、可审计的。APP 引导式"折叠展示"后，**引导过程中间步骤（第 2 步抛问题时）到底查没查教材**，没有可观测信号。
- `08` R3 只说"折叠展示一张精简证据条"——那是**收口时**的。但引导的**中途**（最危险、最容易瞎带处）没有审计点。
- **判语**：必须设计"引导审计日志"——每轮内部 search 调用（关键词+命中行号）落 Supabase，云展开可查、PC 端可抽检。否则"引导质量"是不可观测量，无法回归验证，违背全案"客观信号优于主观自评"的红线。

**架-3 🟡 OCR 是一条全新未设计管线（范围扩张）**
- 拍照→OCR→题干结构化→云确认，是新增组件：OCR 选型（云端 API？成本？中文法律术语识别率？）、错误处理、与"云确认题干"的 UX，`00` 只有一行"要做"。
- `00` 风险 5 已警告 MVP（仪表盘+答疑+背诵+沉淀+同步）单人 3 周偏紧——OCR 再插进首期会进一步挤压。
- **判语**：OCR 价值真实（在职多对纸质卷），但它与"答疑核心管线"正交，应排到**答疑直答版跑通之后**，非首期阻塞项。

**架-4 🟡 沉淀触发与引导状态耦合，边界未定**
- 五步第 5 步沉淀靠触发词"懂了/下一题/下课"。但引导式对话里云可能不说这些词就切下一题；**逃生舱中途放弃**算不算暴露了弱项？引导**卡在第 3 步降档**揭答案——这个"卡点"本身是最该沉淀的弱项信号（`00` 说"每步推导都是诊断信号"），但触发逻辑没覆盖"中途卡点"。
- **判语**：候选弱项的生成时机要从"结束触发词"扩展到"引导过程中的卡点捕获"，否则引导式最大的协同红利（精准抓弱项）漏接。

**架-5 🟡 跨会话记忆未决，削弱引导连续性**
- `00` 风险 1 + C-3 第 14 条：Web≠Claude Code 的那 5% 正是**跨会话记忆需手动管理**（Supabase 存摘要、新对话注入，但"注入多少/防旧结论污染"未决）。
- 引导式**尤其依赖**"记得云上次卡在哪"才能渐进逼问。今天引导到一半、明天新对话失忆 → 引导退回原点，连续性断裂。这对"重引导案例题"（可能跨天）伤害最大。
- **判语**：跨会话记忆方案（注入什么摘要、如何防旧/错结论污染）必须在引导式落地前定，否则引导是"每天重新认识你"的金鱼记忆。

### 2.3 架构师结论
> 答疑数据流确实是两核里更轻的，Tool Use 复刻 grep、复用 `11` 同步都扎实。但**"引导式"这一个交互选择，把原本最简单的一问一答变成了有状态多轮对话机**——它同时抬高了**实现复杂度（状态机不可靠）、单题成本（多轮全价输出）、质量不可观测性（内部预检无审计点）**，再叠加 OCR 新管线、沉淀触发耦合、跨会话记忆未决。**比背诵简单是真的，但"引导式"让它没有想象中那么简单**，这几项必须在动手前显式设计，不能靠"塞进 system prompt 让 Opus 自觉"。

## 三、综合结论（两视角收敛于同一句）

> **答疑系统的"轻"和"快"建立在"一问一答 v2.3 直答版"上（PC 端已验证、数据流最简）；而"引导式"是真正的价值升级，也是全部不确定性的来源——它对症、值得做，但把质量验证、成本、状态复杂度、记忆连续性四件事一起拉高了，且这些恰好都被"答疑很简单"的直觉掩盖了。**

两个视角独立分析，撞到同一结论：**引导式值得做，但不该一上来就全量做。** 先把直答版管线跑通，引导式作为**叠加在交互层的第二迭代**（它不改底层数据/工具/同步，纯交互升级，天然适合分期）。

### 3.1 开工前必须收口（P0）
1. **引导式分型表**【教-1】：明确哪类题引导（理解/案例/易混辨析）、哪类直答（陌生记忆点、纯法条定位）。记忆类陌生点默认直答，不强套苏格拉底。
2. **内部预检可审计**【教-2 + 架-2】：引导每轮的内部 search（关键词+命中行号）落审计日志，折叠可展开、PC 可抽检。引导质量必须是**可观测量**，否则违背全案"客观信号优于主观自评"红线。
3. **引导状态机：显式 or 模型自觉，先定**【架-1】：若靠 system prompt 让 Opus 自管"卡几轮/降档/收口"，先小样测它可靠性；不可靠则需显式状态字段。同时把"重引导多轮"的真实 token 量回灌 `10` 成本模型校验。

### 3.2 范围与里程碑建议
4. **第一期答疑 = v2.3 直答版**（一问一答 + 证据卡 + grep Tool Use + 缓存 + 沉淀），把管线和工具链趟通——这也正是 `09`/`01` P1-8"答疑 1 周可用、且为背诵检测先趟通评分引擎"理由的落地形态。
5. **引导式作第二迭代**叠加：分型表 + 状态机 + 审计日志三件套就绪后再上。它是交互层升级，分期零损失。
6. **OCR 后置**【架-3】：排到直答版之后，非首期阻塞。上线时强制"OCR→云确认题干"一步【教-3】。
7. **跨会话记忆方案本轮定**【架-5】：注入什么摘要、如何防旧/错结论污染——引导式落地的前置依赖。
8. **答疑注入心得做"真题背书过滤"**【教-4】：自测候选/待观察区条目不进引导的已知争点库，防演练数据污染真实学习。

### 3.3 与背诵评估（`09`）的呼应
- `09` 3.3 已埋伏笔："**先做答疑、为背诵检测先趟通评分引擎+grep 工具链**"。本评估给出落地形态：**答疑直答版就是那个趟路者**——它的 grep Tool Use、证据卡评分纪律、沉淀同步，全部是背诵 L2/L3 检测要复用的。
- 因此"答疑直答版优先"同时服务两个目的：① 最快出价值；② 为背诵检测预制评分引擎。这强化了 `09` D-评3"成本实数出来前暂缓定序"里"答疑先做"的一侧。

## 五、跨会话记忆方案（D-答5 收口，引导式连续性的前置依赖）

> 病根（架-5 + `00` 风险1 + C-3 #14）：Web≠Claude Code 那 5% 差距正是**无自动跨会话记忆**。引导式尤其依赖"记得云上次卡在哪"才能渐进逼问；今天引导到一半、明天新对话失忆 → 退回原点。但**简单全量灌历史会带来两害**：① token 爆炸（每次塞全部历史）；② 旧/错结论污染（用过时判断带真实学习）。下面是解法。

### 5.1 三类持久数据的边界（沿用 `11` 三层，不破真相源）
| 类 | 内容 | 主人 | 跨会话怎么用 |
|---|---|---|---|
| **A 资产层** | CLAUDE.md / skill v2.3 / 心得 / 高频考点 / 易混库 | markdown@GitHub | 每次**检索式注入**（grep 相关段，非全量），本就是"知识库"不算"记忆" |
| **B 真实弱项** | `当前弱项.md` 真题背书部分 | markdown（PC 登记） | 新会话注入 **Top5 + 与当前题相关项**，自评/待观察条目**不注入**（防污染，对应教-4） |
| **C 会话记忆** | 答疑过程暴露的卡点 / 上次引导到第几步 | Supabase（状态层，**不回 markdown**） | 本节核心，见 5.2 |

### 5.2 会话摘要 = 结构化字段（不是自由文本，防漂移）
答疑结束触发词（"懂了/下一题/下课"）时，AI 生成一条结构化摘要写 Supabase：
```
session_summary {
  date            -- 日期（驱动 TTL 衰减）
  科目 / 考点ID    -- 锚点，新会话靠它检索（呼应锚点规范化层）
  题型            -- 选择 / 案例 / 简答
  卡点            -- 卡在五步的第几步 + 具体混淆（如"指示交付与占有改定混"）
  状态            -- 未解决 / 已澄清（本会话收口是否验证通过）
  关联弱项ID      -- 若已沉淀为弱项
  TTL标记         -- 衰减/过期标志
}
```
> 用结构化字段而非自由文本：① 可按科目/考点检索（省 token）；② 字段约束防摘要"夹带未核实判断"。

### 5.3 新会话注入策略（检索式，不全量）
1. 云进入答疑 → 贴题/拍题 → **按 科目+考点ID 检索**最近会话摘要。
2. 只注入**与当前题相关的"未解决"卡点**（最多 K 条，建议 K=3-5）+ **Top5 弱项**。
3. 不相关历史**不注入**——省 token（grep=省钱钥匙的同一逻辑）+ 防噪音干扰引导。
4. 引导时 AI 据此衔接："你上次在'指示交付'这步卡过，这道题正好又碰到，先说说……"——连续性恢复。

### 5.4 防污染四条（直接回应 C-3 #14"防错误旧结论污染"）
1. **时效衰减**：摘要带日期，超 X 天（建议 30 天）的旧卡点降权或不注入——云可能早会了，别拿陈年卡点当现状。
2. **状态标记**：卡点标"未解决/已澄清"；已澄清的（后续答疑/检测验证答对）**不再作为弱项注入**，防"系统永远记得你不会"的旧结论压人。
3. **真相源隔离**：会话摘要是**过程记录非权威结论**——引导争点仍以 markdown（真题背书的心得/弱项）为准，摘要只提供"云的个人卡点历史"这**一个维度**，**不得用摘要里的临时判断覆盖 markdown 权威**。
4. **错误结论不沉淀**：遵 v2.3 机制⑥候选隔离——看标答前的临时结论以"？/待核对"开头，**被证伪的永不进摘要**。摘要只记"云在 X 处卡过/已澄清 Y"这类过程事实，不记 AI 自己未核实的判断。

### 5.5 与背诵/弱项管线的复用
- 摘要写 Supabase 走 `11` 同款"待办筐"通道；其中"卡点→候选弱项"经云确认后由 PC 登记员去重并入 `当前弱项.md`（去重键科目+知识点）。
- **会话摘要本身不进 markdown**（它是状态/过程，`11` 已定状态不回 markdown）；只有"确认为弱项"的那部分才升格进 markdown 真相源。边界清晰，不污染内容层。
